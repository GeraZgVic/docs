## 1) Contexto (correo Kueski)

Kueski solicita:
1. **Creación de pago (API)** con parámetros mínimos: `order_id`, `description`, `amount`, `items`, `shipping` (si aplica), `callbacks`.
2. **Merchant response (Webhook / Listener)**: Kueski notifica el estatus final del Loan. El proceso finaliza cuando el merchant recibe y responde correctamente.
3. **Autenticación**: JWT HS256 firmado con `api_secret`, enviado como header `Authorization: Bearer` en la respuesta del webhook.
4. **Widgets** (detalle de producto y resumen de carrito).
5. **Certificación**: 3 pruebas (Success, Denied, Cancelled).

## 2) Webhook — Estado actual ✅

### 2.1 Endpoint DEV

| Parámetro | Valor |
|---|---|
| URL | `https://dev.importacionesamexico.com.mx/index.php?dispatch=kueski.webhook` |
| Método | POST |
| Content-Type | application/json |
| Respuesta | HTTP 200 + header `Authorization: Bearer {JWT}` + body JSON |

> Endpoint disponible públicamente vía HTTPS. Basic Auth deshabilitado en DEV para permitir acceso desde Kueski.

### 2.2 Matriz de respuesta y acciones internas

| Status entrante | Guard | Respuesta | Acción CS-Cart |
|---|---|---|---|
| `approved` | ✅ válido | `accept` | `fn_change_order_status($order_id, 'P')` — Paid |
| `approved` | ❌ inválido | `reject` | Ninguna — solo auditoría |
| `denied` | ✅ válido | `ok` | `fn_change_order_status($order_id, 'D')` — Declined |
| `denied` | ❌ inválido | `ok` | Ninguna — solo auditoría |
| `canceled` | ✅ válido | `ok` | `fn_change_order_status($order_id, 'I')` — Canceled |
| `canceled` | ❌ inválido | `ok` | Ninguna — solo auditoría |
| `cancelled` | ✅/❌ | `ok` | Igual que `canceled` |
| (desconocido) | — | `ok` | Solo registrado con warning en logs |

### 2.3 Guard de seguridad (v1.5)

Validaciones ejecutadas **antes** de cualquier `fn_change_order_status()` o `UPDATE ?:kueski_payments`:

1. `order_id` es numérico y mayor a 0
2. La orden existe en `cscart_orders`
3. Existe mapping válido en `cscart_kueski_payments` con coincidencia de `order_id` y `payment_id`

En cualquier fallo de guard:
- Se guarda evidencia en `cscart_kueski_webhook_events`
- No se modifica ninguna orden
- No se actualiza el mapping interno
- Se responde con JWT válido

### 2.4 Idempotencia

- Evento único definido por la llave: **(payment_id, status)**
- Si llega el mismo evento dos veces, no se duplica el registro, no se vuelve a cambiar el estado de la orden, y se responde con el mismo `response_status`.
- Implementado con `INSERT IGNORE` + `UNIQUE KEY (payment_id, status)`.

### 2.5 JWT (outbound)

- Generado por `JwtService::generateHs256()` en cada respuesta HTTP 200.
- Algoritmo: **HS256**, firmado con `api_secret`.
- Payload: `public_key`, `iat`, `exp` (iat + 300s), `jti` (SHA256 de `api_secret:iat`).
- Emitido como header: `Authorization: Bearer {token}`.
- Credenciales leídas desde settings del addon vía `Registry::get('addons.kueski_pay.*')`.
- Si faltan credenciales: responde HTTP 500 y termina con `exit` (no continúa ejecución).

#### Ejemplo de JWT generado en Sandbox

**Header (Base64Url-decoded):**
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**Payload (Base64Url-decoded):**
```json
{
  "public_key": "f98c02ed-3906-4f00-8ca9-e744d0048131",
  "iat": 1771882197,
  "exp": 1771882497,
  "jti": "sha256(api_secret:iat)"
}
```

## 3) Payment Processor — Estado actual ✅

### 3.1 Archivo
```
app/addons/kueski_pay/payments/kueski_pay.php
```

### 3.2 Flujo implementado
```
Create Payment → Persistir mapping → Redirect al widget → Webhook actualiza orden
```

### 3.3 Lógica principal (POST)

- Lee credenciales desde `Registry::get('addons.kueski_pay.*')`.
- Detecta entorno: `strtolower(trim($mode))` — acepta `sandbox`, `testing`, `production`. Fallback a sandbox si valor inválido.
- Construye payload con `order_id`, `description`, `amount.total` (redondeado con `round(...,2)`), `currency=MXN`, `callbacks` HTTPS.
- `items` solo se incluye en el payload si el array no está vacío.
- Loggea payload completo antes de llamar a Kueski (DEV-only evidence).
- Llama a `KueskiApi->createPayment($payload)`.
- Persiste mapping en `cscart_kueski_payments` con `INSERT ... ON DUPLICATE KEY UPDATE`.
- Redirige al cliente a `callback_url`.

### 3.4 Regla de oro de reintento por status (v1.6)

Aplica en dos puntos: session lock y DB-level idempotency.

| Status en BD | Acción |
|---|---|
| `created` | Reusar `callback_url` — redirect directo |
| `approved` | Redirect a detalle de orden + notice (no recrear pago) |
| `denied` / `canceled` / `cancelled` | Crear nuevo payment — UPSERT sobrescribe mapping |
| (vacío / desconocido) | Permitir flujo normal |

### 3.5 Idempotencia del processor

- **Session lock**: overwrite directo del timestamp (sin patrón `unset + set`).
- **DB check**: reutiliza mapping solo si `status=created`. Si es terminal, crea nuevo payment.

### 3.6 PAYMENT_NOTIFICATION (return/cancel)

- Solo para UX — no finaliza la orden.
- `return`: muestra mensaje "confirmando pago" y redirige a detalle de orden.
- `cancel`: muestra mensaje de cancelación y redirige al checkout.
